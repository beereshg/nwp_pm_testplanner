1. Architecture / Micro-architecture and Functionality

Fast RAPL is a sub-millisecond power limiting mechanism that complements the standard Socket RAPL PL1/PL2 NN-PID control loop. While Socket RAPL PL1/PL2 respond on ~1 ms loop cadence, Fast RAPL responds to transient power excursions at ~500 µs using SVID IMON telemetry and a dedicated FastRAPL PID. The PLR (Performance Limit Reason) bit FAST_RAPL in the CBB TPMI reflects Fast RAPL throttling activity. ⚠️ NWP Status Note: The NWP PM MAS §3 ZBB list includes FastRAPL as ZBB'd ("SIMPL, FastRAPL, Fine Grained Energy"). However, FV team TC inference for the child TCs (22022421939, 22022421944) classifies them as Runnable_On_N-1 citing "Fast RAPL (PEM/UFS feature) is Supported on NWP." Confirm against the latest NWP PM MAS before silicon validation. TCD content is written assuming Fast RAPL is runnable pending confirmation. Key architectural facts (from DMR RAPL Simplification FAS + Primecode Fast RAPL flow + HPM Message Spec):

  • Fast RAPL is a FastRAPL PID evaluated and communicated every 500 µs — all computations and downstream HPM communications run at this cadence
  • It does not use an EWMA low-pass filter at the front end (unlike Socket RAPL PL1/PL2)
  • When instantaneous socket power (measured via SVID IMON) exceeds the Fast RAPL threshold, Fast RAPL computes a frequency ceiling and drives it downstream
  • Threshold is not user-programmable through standard register interfaces — default is conceptually Power_Target ≈ 1.2 × TDP; post-si override hooks exist for debug only
  • The frequency ceiling from Fast RAPL is min'd with all other active RAPL controller outputs → unified RAPL_PERF_LIMIT (HPM opcode 0x14) sent every 500 µs
  • DMR removed the older dedicated Fast RAPL HPM message — Fast RAPL is now folded into the unified RAPL_PERF_LIMIT message with the FAST_RAPL indicator bit
  • If SVID telemetry is unavailable, Fast RAPL cannot evaluate power → drives max/no-effective-limit (0xFF) rather than real throttling
  • Response time (PLR) < 4 ms (vs ~25 ms for PL2 settling)
  • PEM_STATUS[FAST_RAPL] (bit 9) uses RW0C semantics

3. Reset, Power, and Clocking
  • Fast RAPL PEM loop runs on the NIO / IMH-P root die at ~500 µs cadence; initialized during PH6
  • Requires SVID bus operational and IMON telemetry available
  • Fast RAPL threshold is configured by PrimeCode from TPMI / fuse inputs at boot
  • Warm reset re-initializes Fast RAPL state; cold reset resets all RAPL controllers including Fast RAPL
  • BIOS knobs TurboPowerLimitLock and TurboPowerLimitCsrLock must be disabled for runtime testing

---

4. Programming Model
  • Fast RAPL threshold: Not user-programmable through standard register interfaces — configured by PrimeCode from TDP fuse (Power_Target ≈ 1.2 × TDP default); post-si debug override hooks exist but are not product interfaces
  • PEM status: sv.socket0.cbb{0,1}.base.tpmi.pem_status.pem_excursion_bit (bit 9) — RW0C semantics; clear by writing 0; avoid race conditions in reads
  • PLR mailbox: sv.socket0.cbb{0,1}.base.tpmi.plr_mailbox_interface — FAST_RAPL bit set when Fast RAPL is throttle cause
  • RAPL_PERF_LIMIT (HPM 0x14): Unified message carrying FAST_RAPL indicator bit + resolved RAPL_PID_FREQ_OUTPUT + CLOS_*_LIMIT outputs — sent every 500 µs when Fast RAPL is active
  • SVID unavailable fallback: If SVID/IMON telemetry is unavailable, Fast RAPL drives 0xFF (max/no-effective-limit) rather than throttling — verify SVID health before testing
  • Test stimulus: PMX cpu_rapl module with nwp.xml → python runPmx.py -x nwp.xml -p cpu_rapl; or PTU workload to exceed Fast RAPL threshold
  • NWP CBB count: Loop over cbb0, cbb1 only (not range(4))
  • Root die path: sv.socket0.nio.punit.* for root-die TPMI (not imh0)

5. Operational Behavior

Fast RAPL entry flow:

1. NIO PrimeCode 500 µs timer fires; reads SVID IMON for instantaneous socket power

2. FastRAPL PID evaluates: actual power vs. threshold (≈ 1.2 × TDP)

3. If power exceeds threshold: Fast RAPL computes frequency ceiling; RAPL_PERF_LIMIT = min(Fast RAPL, PL1, PL2, ...) updated

4. HPM 0x14 (RAPL_PERF_LIMIT) distributed to CBB0 and CBB1 with FAST_RAPL bit set — every 500 µs (decoupled from 1 ms Socket RAPL subflow)

5. Each CBB PCode applies frequency ceiling; sets FAST_RAPL PLR bit in tpmi.plr_mailbox_interface; updates LEAF_PERF_STATUS via HPM 0x16

Fast RAPL exit flow:

1. Instantaneous power drops below Fast RAPL threshold

2. Next 500 µs timer: FastRAPL PID relaxes ceiling; RAPL_PERF_LIMIT recalculated (dominated by PL1/PL2 ceilings)

3. HPM 0x14 sent without FAST_RAPL being limiting contributor

4. CBB PCode removes Fast RAPL ceiling; FAST_RAPL PLR bit clears; pem_excursion_bit cleared by SW (RW0C)

Response time (PLR): < 4 ms from power excursion to frequency ceiling applied

7. Security / Safety / Policy
  • Fast RAPL is a firmware-level power protection mechanism; threshold configuration is privilege-controlled
  • Test stimulus (PMX cpu_rapl) requires OS ring-0 and PMX framework access
  • Intentional power overload to trigger Fast RAPL must be done in controlled silicon lab environment
  • PEM/PLR register reads require PythonSv / namednodes privileged access
