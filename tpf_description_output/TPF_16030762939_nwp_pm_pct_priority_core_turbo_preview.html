1. Feature Classification — Silicon-Heavy with Firmware Orchestration
PCT (Priority Core Turbo) is a silicon-heavy feature with moderate firmware orchestration.
Core insight: Hardware enforces HP/LP TRL differentiation; firmware/BIOS/PCode configure and expose the capability.
Fuse gate: CAPID4.bit29 = PCT_ENABLE — feature is silicon-capability-gated, not a firmware construct.

Silicon-Heavy Evidence

  • CAPID4.bit29 fuse-gates the entire feature — silicon capability, not firmware
  • HP/LP TRL enforcement is in silicon CLOS hardware (Acode + CBB PCode at RTL level)
  • Core frequency derating for HP vs LP implemented in Acode microcode
  • Key validation scenarios (C6 interaction, TDP convergence, power/frequency) require real silicon
  • PSS model gaps exist: VP cannot model exact Acode behavior; HSLE cannot model cross-die HPM

Firmware Orchestration Responsibilities

Firmware AgentPCT Role
BIOSCapability check (CAPID4.bit29); PCT knob exposure/hiding; TPMI/MSR programming order
PCode (CBB)Dereferences SST_CLOS_CONFIG; applies HP/LP TRL to cores per partition config
PrimeCode (NIO/IMH)Sends HPM RAPL_PERF_LIMIT messages with PCT-adjusted power limits
Acode (core µCode)Applies derating ratio to HP vs LP cores at execution level

5. Why PSS ≠ FV and PSS ≠ PV

PSS vs FV: Time Axis (Pre- vs Post-Silicon)

Case 1 — Firmware bug in PCode TPMI write:
PSS → FAIL ✅ (catches it 6+ months before silicon)
FV → FAIL ✅ (confirms on real silicon)
→ PSS = early detection; FV = ground truth
Case 2 — Silicon TPMI decoder hardware bug:
PSS → PASS ❌ (RTL model may not have same layout bug)
FV → FAIL ✅ (only FV catches it)
→ FV is irreplaceable for silicon bug detection
Case 3 — Model gap (missing TPMI field in VP):
PSS → PASS ❌ (model doesn't implement that field)
FV → ground truth ✅
→ PSS passes silently; only FV reveals the true behavior

PSS vs PV: Interface Axis (Zero Overlap)

  • PSS environments (VP/HSLE/XOS) run minimal boot — no Linux OS, no intel-speed-select driver, no sysfs, no sst tool
  • PV cannot run pre-silicon (requires booted Linux on real silicon)
  • Therefore: PSS ❌ cannot validate software stack; PV ❌ cannot run pre-silicon
  • Conclusion: Zero content overlap between PSS and PV
