Section 1: Validation Scope

Validates the **below-Pm fast_throttle** scenario defined in [TCD 16031169418 -- Socket RAPL - Below-Pm / Fast Throttle](https://hsdes.intel.com/appstore/article-one/#/16031169418) §5, row "Below-Pm fast_throttle".

Environment: NWP post-silicon (SVOS + PythonSV) or virtual platform.

When Socket RAPL PL1 forces the NN-PID resolved frequency output (`RAPL_PID_FREQ_OUTPUT`) below **Pm** (the fused minimum operating ratio), PrimeCode asserts the **fast_throttle** wire to CBB PCode. CBB PCode applies **clock division** and **architectural throttle** on all active cores, causing effective frequency to drop below Pm. When the PID output recovers above Pm, fast_throttle de-asserts and frequency recovers. This TC harasses PL1/PL2 limits and verifies that the throttle/recovery cycle executes correctly on both NWP CBBs.

> Ref: Wave 3 Common HAS [Socket RAPL — Throttling Below Pm](https://docs.intel.com/documents/pm_doc/src/server/Wave3_common/Socket_RAPL/Socket_RAPL.html#throttling-below-pm-spr-onwards)

Section 4: Test Steps

Phase 1: Initialization and Baseline Capture

1. **Import cpu_rapl utilities and read Pm ratio** — PythonSV session imports `diamondrapids.pm.pmutils.cpu_rapl as cr`. Script reads `PM_RATIO` from `pcudata_global_pm_ratio_ia(0)`.

  • Expected: `PM_RATIO` is a valid integer (typically 4–8 for NWP; represents minimum guaranteed ratio × 100 MHz).

2. **Read and save current PL1/PL2 limits** — Script saves `crmsr_pkg_rapl_limit` for restore. Records baseline `package_energy_status` (TPMI) and `pkg_rapl_perf_status`.

  • Expected: PL1 = TDP (fused); PL2 = 1.2 × TDP. Energy status is non-zero and progressing.

3. **Enumerate active cores across both CBBs** — Script iterates `die.core_pmsb.core_pmsbs` and checks `pma_debug2.acp_enabled` for each core.

  • Expected: Active cores identified on CBB0 and CBB1 (NWP: up to 48 cores per CBB, 96 total).

Phase 2: Harass PL1/PL2 to Force Below-Pm Throttle

4. **Start TDP workload** — Launch `ptu -ct 1` or equivalent all-core AVX workload to ensure RAPL is active and power draw approaches TDP.

  • Expected: Cores running at P1 frequency; power draw near TDP.

5. **Harass PL1 — set aggressively low limit via TPMI** — Script programs `socket_rapl_pl1_control.pwr_lim` to a value well below current power draw (forcing PID output below Pm). TPMI interface used (`use_tpmi = 1`).

  • Expected: PrimeCode NN-PID resolves `RAPL_PID_FREQ_OUTPUT < PM_RATIO`; `fast_throttle` wire asserted to both CBBs.

6. **Verify throttle status** (`_VERIFY_THROTTLED_`, bit1) — Script reads per-die PCode state:

  • `socket.imh0.pcudata.socket_rapl_pid_output` — PID output value
  • Per-core `target_wp1.core_frequence` — effective frequency
  • `plr_die_level` — PLR (Performance Limit Reasons) bits
  • Expected:
  • PID output ≤ `PM_RATIO`
  • Core ratios converge to ≈ Pm (within 1 bin tolerance)
  • Mesh ratio converges to Pm: `die.pcudata.min_mesh_ratio`
  • PLR bit set in `plr_die_level` (if not in pcudata, check TPMI PLR)
  • `power_limitation_status` and `power_limitation_log` set in `pkg_therm_status`

7. **Verify counters** (`_VERIFY_COUNTERS_`, bit2) — Script reads:

  • `dfo_primary_plane_rapl_perf_status` — RAPL perf status counter
  • `dfo_pkg_rapl_perf_status` — package RAPL perf status
  • `package_energy_status` (TPMI) — energy accumulation
  • Expected: All counters progressing (non-stale); energy counter incremented from baseline.

8. **Verify TPMI resolved limits match programmed values** — Script cross-checks:

  • `cr.read_programmed_val(sktNum, method="tpmi", limit=1)` → resolved PL1
  • `cr.read_programmed_val(sktNum, method="tpmi", limit=2)` → resolved PL2
  • Expected: Resolved limits match programmed TPMI values (within PL2 0.9× target factor).

9. **Harass PL2** — Script repeats aggressive limit programming for PL2 via TPMI.

  • Expected: Same throttle/counter behavior as PL1 harass.

Phase 3: Recovery and Restore

10. **Restore original PL1/PL2 limits** — Script writes saved baseline values back to TPMI `socket_rapl_pl1_control` and `socket_rapl_pl2_control`.

  • Expected: `fast_throttle` de-asserts; frequency recovers above Pm.

11. **Verify throttle released** — Script waits ~5 seconds, then re-checks throttle status.

  • Expected:
  • Core ratios recover above `PM_RATIO`
  • `fast_throttle` de-asserted
  • Fast RAPL throttling no longer in progress
  • No MCA, no hang during entire test

12. **Poll final coverage registers** — Script invokes `pmx_coverage.poll_all_registers()` for coverage artifacts.

  • Expected: All measured registers captured; test exits cleanly (exit code 0).

Section 5: Pass/Fail Measurement Method

**Bar:** Per [TCD 16031169418 -- Socket RAPL - Below-Pm / Fast Throttle](https://hsdes.intel.com/appstore/article-one/#/16031169418) §5: "fast_throttle asserted when output < Pm; clock div + arch throttle applied; freq drops below Pm; both CBBs affected; recovery when output >= Pm."

**Measurement procedure (this TC):**

# Check Method Pass Criterion
1 Core ratio convergence to Pm Read per-core `target_wp1.core_frequence` and compare to `PM_RATIO` `mean_core_ratio - PM_RATIO ≤ 1` bin (100 MHz)
2 Mesh ratio convergence Read `die.pcudata.min_mesh_ratio` per compute die `mesh_freq - mesh_frequency_current ≤ 1` when at Pm
3 PLR attribution Read `plr_die_level` per compute die (pcudata + TPMI fallback) PLR bit non-zero during throttle
4 RAPL perf status counters `dfo_pkg_rapl_perf_status` and `dfo_primary_plane_rapl_perf_status` Counters progressing (not stale)
5 Energy counter progress `package_energy_status` (TPMI) at start vs end of throttle phase End > Start
6 Power limitation log `crmsr_pkg_therm_status.power_limitation_log` Set during throttle phase
7 TPMI/resolved limit consistency `cr.read_programmed_val()` TPMI vs resolved Match within tolerance
8 Recovery after restore Core ratios, mesh ratios post-restore Recover above PM_RATIO
9 System stability No MCA, no hang, no error exit Script exit code = 0

**Execution assertions (this TC only):**

  • No IERR / MCA during entire test
  • Script completes without `print_and_exit(socket, 999, ...)` error calls
  • `package_energy_status` (TPMI) monotonically increasing throughout test
  • Fast RAPL throttle released after PL1/PL2 restore (log line: "FAST RAPL throttling not in progress")

**FAIL when ANY of:**

  • `mean_core_ratio - PM_RATIO > 1` during below-Pm phase → cores did not converge to Pm
  • `plr_die_level == 0` across all compute dies during throttle → PLR not attributed
  • `package_energy_status` stale (not changing) → energy accounting broken
  • Script exits with non-zero code or `print_and_exit` error
  • MCA or system hang at any point
