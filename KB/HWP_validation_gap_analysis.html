HWP Feature — Review-Ready Validation Gap Analysis

NWP PM · 2026-07-01 · Sources: Core P-state HAS · ACP PM HAS · Current TC coverage (TPF 16030686481)

Scope & Architectural Reference

This gap analysis covers HWP Native and OOB mode validation across the full P-state processing stack:

  1. Request input and resolution — source selection (thread / package / PECI), normalization (min ≤ desired ≤ max), capabilities boundary
  2. APS / UBPS autonomous selection — DESIRED=0, utilization-based ramp, EPP + Activity Window
  3. SAPM-DLL / EET refinement — dynamic EPB resolution, frequency deration factor from scalability
  4. Final clipping by platform constraints — RAPL, Thermal, Turbo limits dominate HWP-requested frequency
⚠️ Architecture note: EET can reduce granted frequency but does not increase it. HWP operates inside the broader platform envelope — RAPL and Thermal always dominate. If ACTIVITY_WINDOW = 0, pCode uses an internal EPP-derived default. EET feeds into Autonomous HWP / LegacyP through the energy-efficiency layer.

Test Plans — Phase 1: Must Close (P1)

TP-HWP-01 — HWP + EET Deration Interaction [Gap G1]
Objective
Verify that when HWP operates in autonomous mode (DESIRED=0), EET derates frequency for non-scalable workloads, and that the deration is reduced for scalable workloads. Proves EET is active in the autonomous HWP path.
Setup
HWP enabled (Native mode); Turbo enabled; EET enabled; at least two workloads: (a) non-scalable / memory-bound / stall-heavy, (b) scalable / compute-bound. Optional A/B with EET disabled for comparison.
Stimulus
Program HWP: MIN=floor, MAX=highest, DESIRED=0, EPP=perf-oriented. Run non-scalable workload → wait for stabilization. Switch to scalable workload. Optionally repeat both with EET disabled.
Observability
Primary: IA32_PERF_STATUS (0x198) — executed ratio / effective frequency. Secondary: HWP debug telemetry if available. Optional: EET-disabled A/B comparison.
Pass Criteria
(1) Non-scalable workload shows lower granted frequency than scalable workload under identical HWP settings. (2) EET-disabled case delivers higher frequency than EET-enabled for non-scalable workload. (3) EET does not increase frequency above APS request.
TP-HWP-02 — HWP + RAPL Power Limit Interaction [Gap G2]
Objective
Verify that platform power limits (RAPL PL1/PL2) cap HWP-requested frequency. HWP cannot bypass the power envelope. NWP: use TPMI SOCKET_RAPL_PL1_CONTROL — MSR 0x610/0x638 deprecated.
Setup
HWP enabled (Native mode); workload capable of driving frequency toward P0; TPMI RAPL PL1 accessible via sv.socket0.nio.punit.ptpcfsms.ptpcfsms.socket_rapl_pl1_control.
Stimulus
(1) Set HWP: MAX=highest, DESIRED=highest or DESIRED=0 + perf EPP. Confirm freq at P0. (2) Apply aggressive RAPL PL1 (50% TDP). Wait for convergence under same workload. (3) Remove limit → observe freq restoration.
Observability
Primary: IA32_PERF_STATUS. Secondary: RAPL PLR bit in perf_limit_reasons. Optional: workload throughput delta.
Pass Criteria
(1) RAPL limit causes freq to drop below pre-limit level. (2) PLR RAPL bit set during throttle. (3) Removing limit restores freq. (4) HWP request stays active but delivered perf is clipped.
TP-HWP-03 — HWP + Thermal Throttle Cross-Product [Gap G3]
Objective
Verify that thermal protection overrides HWP-requested performance. Delivered frequency must be reduced appropriately under thermal throttle regardless of HWP request.
Setup
HWP enabled (Native mode); thermal trigger available via td.set_dts_temp() injection or controlled heating; high-power workload.
Stimulus
(1) Program HWP for high performance; observe stable high freq. (2) Assert thermal condition (DTS injection above TCC threshold) while workload continues. (3) Remove thermal condition → observe recovery.
Observability
Primary: IA32_PERF_STATUS. Secondary: TCC status; thermal PLR bit in perf_limit_reasons. Optional: package temperature telemetry.
Pass Criteria
(1) Thermal event causes freq to drop below pre-throttle level. (2) HWP does not hold freq above thermal safety policy. (3) On deassertion, freq returns toward original allowed level.
TP-HWP-04 — HWP + C-State Entry/Exit Stability [Gap G4]
Objective
Verify HWP request state is preserved across CC1/CC6 transitions and post-exit frequency is correct. Especially important for autonomous mode where APS depends on recent utilization history.
Setup
HWP enabled (Native mode), DESIRED=0; ability to induce repeated idle/wake cycles; workload alternates active and idle phases.
Stimulus
(1) Program HWP autonomous; drive workload → record steady-state freq. (2) Transition to CC1 or CC6 via controlled idle. (3) Exit idle; resume workload. (4) Repeat 10+ cycles; vary Activity Window setting.
Observability
Primary: IA32_PERF_STATUS. Secondary: C-state residency / entry-exit counters; HWP interrupt status if relevant. Optional: latency to re-converge after exit.
Pass Criteria
(1) No corruption or loss of HWP mode across idle transitions. (2) Freq after exit consistent with configured HWP policy. (3) No stuck-low / stuck-high after repeated cycles. (4) Autonomous mode re-converges correctly.

Test Plans — Phase 2: Robustness & Compliance (P2)

TP-HWP-05 — Activity Window = 0 Default Resolution [Gap G5]
Objective
Verify that ACTIVITY_WINDOW=0 causes pCode to use an internal EPP-derived default rather than treating zero as a literal zero-width window.
Stimulus
(1) Program ACTIVITY_WINDOW=0, EPP=low (perf); run bursty workload → measure responsiveness. (2) Program ACTIVITY_WINDOW=0, EPP=high (energy saving); run same workload → compare.
Pass Criteria
Behavior with ACTIVITY_WINDOW=0 differs with EPP change; results consistent with an EPP-derived internal default (not a literal zero-width window).
TP-HWP-06 — Out-of-Range Request Clamping [Gap G6]
Objective
Verify out-of-range HWP request fields are shaped/clamped safely; final effective values stay within HWP capability bounds.
Stimulus
Read capability bounds. Program: (a) DESIRED > HIGHEST, (b) MIN > MAX, (c) MIN < LOWEST. Run stable workload after each case.
Pass Criteria
Delivered operating point within capability bounds; no illegal / unstable / undefined behavior; request shaping graceful and deterministic.
TP-HWP-07 — HWP EPP to SAPM-DLL Resolved EPB [Gap G7]
Objective
Verify that EPP policy programmed via HWP influences SAPM-DLL EPB-resolution and is not lost before energy-efficiency processing.
Stimulus
(1) HWP fixed min/max/desired; set EPP=low → capture resolved EPB observability and freq. (2) Set EPP=high → same workload → compare.
Pass Criteria
Changing HWP EPP changes resolved EPB behavior in expected direction; downstream freq changes consistently with EPB policy shift.
TP-HWP-08 — HWP + Turbo Disabled (In-Band) [Gap G8]
Objective
