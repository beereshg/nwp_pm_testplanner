## Section 1: Architecture / Micro-architecture and Functionality

**NearTDP** is a platform power qualification feature that stress-tests the SoC by iteratively increasing FIVR voltage offsets across core, fabric, memory, and IO domains until total package power approaches TDP. It is available exclusively on **sample (non-production) parts** gated by the `NEAR_TDP_EN` fuse. The test is driven by the PTU (Platform Tuning Utility) or SSMD tool via the **TPMI OS Mailbox** (package-scoped uCR), serviced by PCode running on the root die.

### Feature Overview

| Property | Detail |
|----------|--------|
| **Purpose** | Push SoC power to TDP boundary to validate thermal cooling and power delivery at rated limits |
| **Availability** | Sample parts only (`NEAR_TDP_EN` fuse = 1); blocked on production parts, TXT-enabled configs |
| **Interface** | TPMI OS Mailbox: `OS_MAILBOX_DATA` + `OS_MAILBOX_INTERFACE` registers, `RUN_BUSY` trigger |
| **FIVR Domains** | Core (VCCCORE), Fabric (VCCCFC, VCCCFCCAB, VCCCFCIO_E/W), Memory (VCCCFCMEM_E/W), IO |
| **Voltage step** | Iterative ~5 mV steps per domain; PCode clips offsets at per-domain hard-coded maximums |
| **Termination** | Power >= 98% TDP, strike count exceeded (no power increase), or voltage limit reached |
| **Security gate** | `NEAR_TDP_EN` fuse + `FLEX_RATIO[OC_LOCK]` + `PCU_MISC_CONFIG[TXT_ENABLE]` checks |

### PCode NearTDP Flow

1. PTU/SSMD writes voltage offset command to `OS_MAILBOX_DATA`; sets `OS_MAILBOX_INTERFACE.RUN_BUSY`
2. PCode (root die) receives mailbox interrupt, validates authorization:
   - Checks `NEAR_TDP_EN` fuse == 1
   - Checks `FLEX_RATIO[OC_LOCK]` == 0
   - Checks `PCU_MISC_CONFIG[TXT_ENABLE]` == 0
3. If authorized: applies voltage offset to requested FIVR domains while enforcing per-domain max limits
4. PCode enforces voltage ordering constraints (e.g., VCCHDC >= VCCCFC)
5. PCode writes back completion status to mailbox; updates `OVERCLOCK_SECURE_STATUS[61]` (OC/NearTDP utilized)
6. PTU measures package power; if below ~98% TDP, repeats with incremented offset
7. Loop terminates on power threshold, strike count, or voltage cap

### TPMI OS Mailbox Interface

```
  PTU / SSMD tool
       |  Write offset + domain → OS_MAILBOX_DATA
       |  Set RUN_BUSY → OS_MAILBOX_INTERFACE
       v
  PCode mailbox handler (root die fast path)
       |  Authorize: NEAR_TDP_EN + OC_LOCK + TXT checks
       v
  FIVR voltage raised on targeted domain(s)
       |  Core FIVR (VCCCORE)
       |  Fabric FIVRs (VCCCFC, VCCCFCCAB, VCCCFCIO_E/W)
       |  Memory FIVRs (VCCCFCMEM_E/W)
       v
  Package power → approaches TDP
       |  If power > PL1: RAPL PL1 throttle activates
       |  Core frequency reduced; RAPL and NearTDP independent
       v
  Test completes: voltage offsets cleared; system returns to baseline
```

### Key Registers

| Register | Access | Description |
|----------|--------|-------------|
| `OS_MAILBOX_DATA` | RW (TPMI SRAM) | Voltage offset value and domain selector for NearTDP command |
| `OS_MAILBOX_INTERFACE` | RW (TPMI SRAM) | Command register; `RUN_BUSY` bit triggers PCode servicing |
| `OVERCLOCK_SECURE_STATUS[63]` | RO | Safe/Unsafe status -- NearTDP allowed on this part |
| `OVERCLOCK_SECURE_STATUS[62]` | RO | Undervolt protection active |
| `OVERCLOCK_SECURE_STATUS[61]` | RO | OC/NearTDP was utilized (set after first valid command) |
| `FLEX_RATIO[OC_LOCK]` | RW (BIOS) | If set by BIOS/TXT, OC/NearTDP commands are rejected |
| `PCU_MISC_CONFIG[TXT_ENABLE]` | RO | TXT enabled -- blocks NearTDP |
| `NEAR_TDP_EN` fuse | RO | 1 = sample part, NearTDP allowed; 0 = production, blocked |

### RAPL Interaction

During NearTDP testing, RAPL power limits remain active unless explicitly raised to TDP or disabled:
- If package power exceeds `RAPL_PL1`, PCode frequency-throttles cores to enforce the limit
- This can **prevent power from reaching TDP** unless PL1 is set to TDP or higher
- Validation best practice: set PL1 = TDP before running NearTDP to avoid premature throttle

### NWP-Specific Deltas

- **PTU tool** is the primary driver on NWP (replaces SSMD from DMR); same TPMI OS Mailbox protocol
- **2 CBBs** (cbb0 + cbb1) -- FIVR domains span both CBBs; PCode applies offsets package-scoped
- **Sample parts only** -- NWP bring-up parts carry `NEAR_TDP_EN` fuse; production SKUs do not
- **Virtual platform environment** -- TC 22022422440 targets virtual_platform; real power measurement is simulation only
- **RAPL PL1 set point**: Must be at or above TDP for accurate NearTDP sweep without premature throttle

## Section 3: Reset / Power / Clocking

- **PH5 (PrimeCode init)**: `NEAR_TDP_EN` fuse read; `OVERCLOCK_SECURE_STATUS[63]` (safe/unsafe) programmed
- **CPL0 (BIOS pre-OS)**: If TXT active, BIOS sets `FLEX_RATIO[OC_LOCK]` to block NearTDP/OC commands
- **OS runtime**: PTU tool writes OS Mailbox command; PCode services on each write
- **Post-test**: PCode clears applied voltage offsets; all FIVR domains return to nominal settings
- **Cold reset**: Voltage offsets cleared; `OVERCLOCK_SECURE_STATUS[61]` cleared; fresh NearTDP sweep needed
- **Warm reset**: State similar to cold reset for NearTDP offsets -- BIOS must re-validate TXT/OC_LOCK state

## Section 4: Programming Model

### Check NearTDP Authorization

```python
nio = sv.socket0.nio0
# Check if NearTDP is allowed on this part
oc_status = nio.tpmi.overclock_secure_status
safe = (oc_status >> 63) & 1
uv_protection = (oc_status >> 62) & 1
oc_utilized = (oc_status >> 61) & 1
print(f"NearTDP safe: {safe}, UV protection: {uv_protection}, OC utilized: {oc_utilized}")
assert safe == 1, "NearTDP not authorized on this part (production fuse or TXT active)"
```

### Set RAPL PL1 to TDP Before Test

```python
import msr
# Set PL1 = TDP, PL2 = TDP (avoid premature throttle during NearTDP sweep)
# Read TDP from TPMI
tdp_watts = nio.tpmi.package_power_info.tdp  # RO field
# Set RAPL PL1 = TDP
# IA32_PKG_POWER_LIMIT (0x610): bits [14:0] = power limit in 1/8W units, bit[15] = enable
pl_units = nio.tpmi.package_power_sku_unit.pwr_unit  # 1/2^pwr_unit W per step
pl1_raw = int(tdp_watts * (2 ** pl_units))
msr.write(0x610, (1 << 15) | (pl1_raw & 0x7FFF))  # PL1 enable + TDP value
```

### Verify OVERCLOCK_SECURE_STATUS After NearTDP Run

```python
# After PTU completes a NearTDP sweep, bit[61] should be set
oc_status = nio.tpmi.overclock_secure_status
oc_utilized = (oc_status >> 61) & 1
assert oc_utilized == 1, "NearTDP/OC mailbox was never utilized"
print("NearTDP command was successfully exercised")
```

## Section 5: Operational Behavior

| Scenario | Expected Behavior |
|----------|-------------------|
| NearTDP on sample part (`NEAR_TDP_EN`=1), TXT disabled | OS Mailbox command accepted; voltage raised; power ramps toward TDP |
| NearTDP on production part (`NEAR_TDP_EN`=0) | Mailbox command rejected; `OVERCLOCK_SECURE_STATUS[63]`=0 (unsafe) |
| TXT enabled (`TXT_ENABLE`=1 or `OC_LOCK`=1) | Mailbox rejected regardless of fuse |
| RAPL PL1 < TDP during test | Power throttled before TDP; NearTDP sweep terminates at PL1 limit |
| RAPL PL1 >= TDP during test | Sweep reaches TDP boundary; throttle only if thermal limit exceeded |
| Voltage offset hits per-domain max (PCode clip) | PCode stops incrementing; returns max-reached status to tool |
| Strike count exceeded (no power increase) | Tool terminates; last applied offset is nominal `OVERCLOCK_SECURE_STATUS[61]` set |
| System recovery post-test | PTU clears offsets; all FIVRs return to nominal; no persistent voltage change |
| MCA / thermal runaway during test | FAIL: system should not crash under NearTDP; indicates cooling or power delivery gap |

## Section 6: Corner Cases & Error Handling

- **TXT runtime activation**: If TXT is activated after NearTDP offsets are already applied, PCode must revert offsets and set `OC_LOCK`; verify no residual offset remains.
- **Simultaneous RAPL throttle**: If RAPL throttles during sweep (PL1 < TDP), verify the mailbox still responds correctly and voltage offsets are not corrupted.
- **Voltage domain ordering violation**: PCode must maintain VCCHDC >= VCCCFC even when PTU requests offset on VCCCFC; verify ordering check is enforced.
- **Cold reset mid-sweep**: Verify all voltage offsets are cleared and `OVERCLOCK_SECURE_STATUS[61]` reset after cold reset; no sticky offsets.
- **Virtual platform limitations**: Real power measurement not possible in VP; TC 22022422440 uses VP environment and can validate mailbox dispatch/register behavior only -- not actual TDP proximity.
- **Multiple mailbox calls in rapid succession**: Verify PCode serializes `RUN_BUSY` correctly; no lost commands.
- **Strike count boundary**: Verify the tool detects and exits correctly when power increase stalls; no infinite loop on clipped offset.

## Section 7: Security / Safety / Policy

- **Sample part gate**: `NEAR_TDP_EN` fuse is the primary security gate -- production parts block NearTDP unconditionally. This prevents voltage manipulation attacks on production systems.
- **TXT/SGX guard**: `FLEX_RATIO[OC_LOCK]` and `PCU_MISC_CONFIG[TXT_ENABLE]` provide defense-in-depth for trusted execution environments.
- **Voltage ceiling enforcement**: PCode enforces per-domain voltage hard limits regardless of offset requested; tool cannot force voltages above safe maximums.
- **UV protection status**: `OVERCLOCK_SECURE_STATUS[62]` indicates undervolt protection is active; NearTDP must not create UV conditions that could compromise secure keys.
- **Ring-0 access only**: OS Mailbox via TPMI is accessible at supervisor privilege; user-mode code cannot trigger NearTDP.
