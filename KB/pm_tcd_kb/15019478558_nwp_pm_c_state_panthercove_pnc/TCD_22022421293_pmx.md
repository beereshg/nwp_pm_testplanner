## Section 1: Architecture / Micro-architecture and Functionality

**PMX (Power Management Extensions)** is a PCode subsystem on NWP (PantherCove PNC) that provides hardware-accelerated PM telemetry and policy enforcement. PMX **C-state check** validates that PMX counters, state registers, and telemetry mechanisms correctly track C-state transitions and report accurate data to software tools.

### T2 Boundary Notes (2026-07-19)

- This TCD owns telemetry consistency only (PMX/PMT view vs architectural counters).
- Functional ownership for entry/exit/residency behavior remains with functional C-state TCDs.

### Key PMX C-State Registers

| Register | Description | NWP Path |
|----------|-------------|---------|
| `pmx_core_cstate` | Current PMX-reported C-state per core | `cbb0.pmc.pmx_core_cstate` |
| `pmx_pkg_cstate` | Package-level C-state per PMX | `uncore.pmc.pmx_pkg_cstate` |
| `pmx_cstate_residency` | PMX-measured C-state dwell | `cbb0.pmc.pmx_cstate_res` |


### NWP-Specific Deltas

| Aspect | DMR (Reference) | NWP (PantherCove PNC) | Impact on Test |
|--------|----------------|----------------------|----------------|
| CBB count | Up to 4 | **2** | All-core loops: `range(4)` -> `range(2)` |
| Cores per CBB | 64 | **48** | Per-CBB loops: `range(64)` -> `range(48)` |
| Total cores | 256 | **96** | Scale workload and verification accordingly |
| PkgC6 | Supported | **ZBB (Zero Bit Budget)** | `IA32_PKG_C6_RESIDENCY` (0x3F9) must stay 0 |
| Register prefix | `cbb{0..3}` | **`cbb{0,1}`** | Adjust all PythonSV paths |
| DCM count | 32 per socket | **12 per socket** | MC6 module loops: `range(32)` -> `range(12)` |
| HW Thread count | 2 per core | **2 per core** | No change |


## Section 2: Interfaces and Protocols

PMX exposes telemetry via:
- **PythonSV** register reads: `sv.socket0.cbb0.pmc.*`
- **TPMI** (Topology Power Management Interface): OS-level PMX register access
- **Perf subsystem**: Linux `perf stat` can expose PMX C-state counters

## Section 3: Reset, Power, and Clocking

- PMX counters are reset on platform reset
- PMX telemetry must be valid within 1 ms of C-state exit (PMX readback latency budget)

## Section 4: Programming Model

```python
# PMX C-state check: verify PMX reports match MSR residency
s = sv.socket0

# Read PMX C-state status for all cores
for c in range(2):
    for k in range(48):
        try:
            pmx_state = s.getbypath(f"cbb{c}.pmc.pmx_core_cstate_{k}").read()
            msr_res = s.getbypath(f"cbb{c}.compute0.module0.core{k}.msr.ia32_c6_residency").read()
            print(f"CBB{c} core{k}: PMX_state={pmx_state:#x} MSR_C6_res={msr_res:#x}")
        except Exception as e:
            print(f"CBB{c} core{k}: {e}")
```

## Section 6: Corner Cases and Error Handling

- **PMX vs MSR mismatch**: If PMX reports C6 but MSR residency shows no increment (or vice versa), there is a PMX calibration bug
- **PMX hang in C-state**: PMX should remain functional during core C6 — verify PMX register access not gated by core power

## Section 7: Security / Safety / Policy

- PMX telemetry access via TPMI should be restricted to privileged (ring-0) software only
