ACP (Acode) vs pCode Ownership • Algorithm Flow • Interface Boundaries • Source: ACP PM HAS + KB articles 2026-07

What is the P-State Stack?

The P-state stack is the complete chain of algorithms and constraints that determine the final core operating frequency on every evaluation cycle. It is not just HWP and not just legacy P-state control — it is the full layered path from request generation to V/F application.

The ACP PM HAS defines it as comprising 9 functional groups, owned by either Acode (ACP-local, per-core) or pCode/SoC (global, slow constraints):

  • Request layer: Legacy P-state, HWP Native, HWP PECI Override
  • Autonomous selection layer: UBPS / APS
  • Efficiency refinement layer: SAPM-DLL, EET
  • Limit/constraint layer: GPSS, TRL/FACT, WP Calc
  • Apply layer: AGV (Acode-initiated GV sequence)
Acode (ACP) owns — local, per-core dynamic computation: thread request processing, resolved thread/core request, APS, SAPM-DLL, EET, local clipping against provided ceilings/floors, AGV initiation.
pCode / SoC owns — global slow constraints: PECI/PKG/BIOS global source resolution, TRL/FACT tables, WP4 row writes, slow-limit resolution into WP1.Fceil and F_floor.

Key architectural principle: ACP/Acode computes the local requested operating point; pCode provides the global envelope and slow constraints.

Legacy P-stateAcodeSame as LegacyAcode processes thread_reqs, computes resolved thread_req and core_req. Uses freq_deration factor from EET. Exposes resolved P_req to SoC.
HWP NativeAcodeSame as Legacy (with P04 guardrail)Acode owns HWP. Processes OS thread req (min, max, desired, EPP, AW). Computes thread & core min/max. Computes resolved thread_req and core_req. Writes Thread0 MSR (Min, Max, Req, EPB). Exposes "core efficiency" metric to SoC.
HWP PECI OverrideSharedSame as LegacypCode handles PECI write, broadcasts to HWP Global CCP CR. pCode resolves all global sources (PKG, PECI, BIOS) into single value. Acode reads HWP_REQ + HWP_GLOBAL to calculate min, max, req, EPP.
UBPS / APSAcodeSame as LegacyAPS runs in Acode Slowloop. Processes resolved thread EPP, thread/core C0 cycles, ubps_gain, ratio_decay, AW. Computes thread/core utilization. Calculates aps_thread_ratio[t] as thread_req. Output of APS is input to HWP — no visible outputs to CCP.
EETAcodeSame as LegacyEET moves to Acode Slowloop. Processes core stall count, L1/L2/L3 stall cycles, SQ Occupancy. Computes avg core scalability via table lookup. Computes freq_deration_factor[c] using resolved EPB from SAPM-DLL. freq_deration_factor[c] is input to Autonomous HWP / Legacy P-state.
SAPM-DLLAcodeSame as LegacySAPM-DLL moves into Acode. Processes thread C0 cycles and resolved thread_req. Computes avg_thread_c0 / avg_thread_c0p0 cycles. Uses ratio and thread_epb[t] to compute resolved_thread_epb and resolved_core_epb = min(thread_epb). Exposes resolved_core_epb to SoC.
GPSSAcode (opt)Same as LegacyGPSS will not exist at global level — only slow limits resolve into WP1.Fceil or F_floor CR. ACP may or may not implement GPSS — IP decision.
TRL / FACTpCodeResolved using WP4TRL and FACT tables remain with pCode. pCode is unaware of Core License. pCode looks at #ActiveCCPs and #HP/LP cores, takes Max(TRL, FACT) rows, sends entire row as WP4 write to all HP/LP cores.
WP CalcSharedSame as LegacyAll global slow limits reside in pCode. pCode writes WP1.Fceil and F_floor. Acode clips resolved core_req[c] against WP1.Fceil, WP4 limits, F_floor. Acode needs to know which algorithms can override Fceil/F_floor (e.g. TRL, IccMax). Acode initiates AGV to apply V&F corresponding to resolved_core_req[c].

Layer B — Autonomous Selection (DESIRED = 0 path)

Layer C — Efficiency Refinement

Resolves EPB context and optionally derates the core request based on workload scalability. EET can REDUCE frequency — never increase it.

Layer D — Limit / Constraint

Global slow limits from pCode bound the locally-computed request. Acode must clip its resolved_core_req against these before initiating AGV.

Layer E — Apply V/F

Practical Interpretation & Validation Guidance

What each layer answers

LayerQuestion it answersOwnerNWP Gap
Legacy P-state / HWP Native"What did the OS ask for?"Acode✓ Well covered
PECI Override"Did an external agent override the OS request?"pCode+Acode✓ Well covered
APS / UBPS"What frequency should autonomous HWP autonomously target?"Acode✓ Well covered
SAPM-DLL"Should internal EPB switch toward performance mode?"Acode✗ G7 — not tested with HWP active
EET"Given scalability and EPB, should we derate frequency?"Acode✗ G1 — not tested with HWP active
WP Calc / TRL / FACT"What is the legal maximum frequency the platform allows?"pCode + Acode clips✗ G2 RAPL, G3 Thermal, G4 C-state
AGV"Apply the final operating point."Acode✓ Not a gap (execution, not selection)

Effective order

APS selects autonomous target (Layer B)
  → SAPM-DLL resolves bias context (Layer C)
    → EET computes deration factor (Layer C)
      → pCode slow limits clip to WP1.Fceil / WP4 / F_floor (Layer D)
        → Acode clips resolved_core_req and initiates AGV (Layer E)
          → Final ratio applied via Core FIVR + PLL
