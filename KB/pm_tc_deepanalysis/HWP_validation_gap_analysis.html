HWP Feature Architecture (End-to-End Stack)

HWP is a hardware-managed P-state mechanism with three layers of processing before a frequency is applied:

Layer 1 — Request Input & Resolution: SW writes HWP request fields (Min, Max, Desired, EPP, Activity Window, Package Control). PM logic resolves source: Thread → Package (if PACKAGE_CONTROL=1) → PECI override. Request is normalized: min ≤ desired ≤ max; DESIRED=0 triggers APS; EPP derived from EPB if not explicit.
Layer 2 — Autonomous Selection (APS/UBPS): When DESIRED=0, UBPS algorithm converts utilization history + EPP + Activity Window into aps_thread_ratio. Gain term ramps up on rising utilization; decay term ramps down on falling utilization. Short Activity Window = responsive; Long = smooth.
Layer 3 — Energy Efficiency Refinement (SAPM-DLL + EET): SAPM-DLL dynamically adjusts EPB based on thread activity. EET computes a frequency deration factor from stall/scalability behavior using SAPM-DLL's resolved EPB. EET can only reduce frequency, never increase it.
Layer 4 — Final Clipping: Acode/pCode clips resolved request against WP1.Fceil, WP4 limits, F_floor. RAPL, Thermal, and Turbo policies remain in pCode and constrain the final frequency. HWP operates within this envelope, not around it.

Validation Gap Summary

🔴 CRITICAL GAP: HWP + EET Interaction
No TC verifies that EET deration is visible when HWP is in autonomous mode (DESIRED=0) and workload is non-scalable. This is a fundamental HWP+EET cross-product that the feature spec describes but the test plan does not cover. EET can silently clip HWP-requested frequencies for efficiency reasons — if broken, HWP users see unexpected lower-than-requested frequencies with no clear indication.
🔴 CRITICAL GAP: HWP + RAPL Interaction
No TC validates that RAPL (Socket PL1/PL2) correctly limits frequency when HWP is active. If RAPL and HWP conflict, RAPL should win. This path is critical for power capping correctness and must be tested explicitly. Note: NWP MSR 0x610/0x638 deprecated — must use TPMI RAPL path.
🔴 CRITICAL GAP: HWP + Thermal Throttle
No TC validates that DTS/TCC-based thermal throttle correctly overrides HWP-requested frequencies. HWP should not prevent thermal protection from capping frequency when temperature exceeds threshold.
🔴 CRITICAL GAP: HWP + C-State Cross-Product
No TC validates HWP state behavior during CC1/CC6 entry and exit. HWP state should be preserved correctly, and frequency should ramp back correctly after C-state exit. This is particularly important for APS mode where utilization history may be stale after deep idle.
⚠️ Minor Gap: Activity Window = 0 Edge Case
No TC explicitly verifies that when Activity Window field = 0, the PM uses an EPP-derived internal default. This is an edge case per the HAS but worth a targeted check.
⚠️ Minor Gap: Out-of-Range Request Clamping
No TC verifies behavior when HWP request fields are set outside HWP_CAPABILITIES range (e.g., desired > highest_perf). The spec says min ≤ desired ≤ max must be enforced — validate that out-of-range inputs are clamped gracefully.
⚠️ Minor Gap: EPP → SAPM-DLL Cross-Check
No TC explicitly reads the SAPM-DLL resolved EPB and verifies it changed as expected when OS writes a new EPP value in HWP mode.
⚠️ Minor Gap: HWP + Turbo Disabled (In-Band)
22022422337 covers OOB+Turbo disabled. A parallel in-band mode test (turbo disabled via MSR while using in-band HWP request) is missing.
