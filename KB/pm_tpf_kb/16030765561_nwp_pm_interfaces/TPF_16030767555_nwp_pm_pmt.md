## Section 1: Feature Classification & Introduction

**PMT (Platform Monitoring Technology)** is Intel's framework for exposing platform telemetry (power, thermal, performance, error, and environmental data) to software consumers such as the OS and BMC on NWP. PMT provides a unified, scalable, and software-enumerable telemetry infrastructure supporting both in-band (IB) and out-of-band (OOB) access paths.

**Classification**: Hardware/Firmware hybrid. PMT objects (Aggregators and Watchers) are hosted on various IPs (iMH PUNIT, OOBMSM, CBB PUNIT). PrimeCode firmware populates telemetry counters. TPMI provides the discovery mechanism. OS/BMC accesses telemetry via MMIO or MCTP.

**Gating mechanism**: PMT is **always enabled** when OOBMSM and TPMI are functional. Individual telemetry endpoints may be gated by feature fuses. PMT discovery is via TPMI PFS (TPMI_ID for PMT aggregator).

**NWP scope**: NWP implements PMT across multiple domains — iMH PUNIT, OOBMSM MSM Aggregator, and CBB PUNIT. PMT supports IB access (OS via MMIO) and OOB access (BMC via PCIe VDM / I3C / MCTP). Key telemetry: package power, core power, temperature, frequency, throttle status, energy counters.

## Section 3: Validation Strategy

PMT validation requires multiple tiers because the feature spans silicon sensors, firmware telemetry population, aggregator engine, TPMI discovery, and OS/BMC consumer layers.

> Layer coverage is mapped in §2 — Validation-Tier Layer Claim table identifies which tier validates each stack layer and flags any unclaimed layers as accepted gaps.

### Tier Definitions

| Tier | Environment | Interface | What it validates |
|---|---|---|---|
| PSS — VP | Simics (virtual platform) | PythonSV → PMT model | FW telemetry population logic, counter update |
| PSS — HSLE | Single-die RTL | PythonSV → PMT RTL | Aggregator HW, SRAM write/read, TPMI access |
| PSS — XOS | Both-die RTL (IMH+CBB) | PythonSV → full RTL | Cross-die PMT aggregation |
| FV | Post-silicon NWP | PythonSV → namednodes | Real sensors + FW + TPMI discovery |
| PV | Post-silicon NWP + Ubuntu/BMC | intel_pmt driver / BMC PLDM | End-to-end telemetry E2E |

## Section 5: Risks & Dependencies

### Active Risks

- **Sensor accuracy dependency**: PMT power/thermal counters depend on SVID IMON and DTS accuracy; sensor errors propagate to telemetry.
- **Aggregator GUID versioning**: If PMT GUID changes between steppings, OS driver may fail to discover telemetry.
- **OOB transport latency**: BMC PMT access via MCTP adds latency; stale data risk for fast-changing counters.
- **TPMI model gap**: VP (Simics) has limited PMT model; aggregator discovery testing requires HSLE or silicon.

### Accepted Coverage Limitations

| Gap ID | Description | Coverage Today | Accepted Rationale |
|--------|-------------|----------------|-------------------|
| **G-1** | Real sensor accuracy (DTS, IMON) | FV + PV only | Requires real silicon sensors |
| **G-2** | OS intel_pmt driver sysfs exposure | PV only | Requires booted Linux + driver |
| **G-3** | BMC OOB PMT via PLDM | PV only | Requires real BMC + MCTP transport |

## Section 6: DFX Considerations

- **PMT SRAM dump**: PythonSV can read all PMT aggregator counters via `sv.socket0.nio.punit.tpmi.pmt_*`
- **Telemetry correlation**: PMT power values should correlate with RAPL ENERGY_STATUS and PEM excursion state.
- **VISA capture**: PMT aggregator SRAM writes can be observed via ITH T2 on OOBMSM domain.
- **turbostat cross-check**: Compare PMT power with turbostat readings for consistency validation.
