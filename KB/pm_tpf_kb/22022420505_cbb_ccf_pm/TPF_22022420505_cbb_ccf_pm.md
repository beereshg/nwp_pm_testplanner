## Section 1: Feature Classification & Introduction

**CBB CCF Power Management** is the comprehensive power management subsystem for the CBB Coherent Compute Fabric (CCF) ring on NWP (Newport). The CCF ring is the primary on-die interconnect within each CBB die, connecting 48 cores (via CBOs), SBOs, and other uncore agents. CCF PM manages ring frequency, voltage, idle states, and cross-die coordination — all with the goal of minimizing power when the fabric is underutilized while maintaining performance when demand is high.

**Classification:** Silicon-heavy + Firmware-heavy hybrid. The CCF ring GV hardware (GVFSM, PLL/FLL, FIVR) is silicon, while the frequency policy, telemetry processing, and cross-die HPM coordination are firmware (CBB PCode for local decisions, IMH/NIO Primecode for cross-die coordination).

**Gating mechanism:** CCF PM is always enabled at boot. Feature sub-modes are controlled by:
- **Fuses:** VF curve points (Pm..P0), max/min ratio caps, NonAutoGV default
- **BIOS knobs:** `UFS_CONTROL` fields (MAX_RATIO, MIN_RATIO, ELC thresholds, THROTTLE_MODE, UNIFORM_CBB_FABRIC_FREQ_MODE) programmed via TPMI during TPMI_INIT PH1.x
- **Runtime TPMI:** OS/PythonSV can modify `UFS_CONTROL` at runtime (subject to BIOS lock)

### NonAutoGV Mode (POR)

NonAutoGV is the only POR frequency management mode for the CBB ring. CCF does not autonomously select V/F — it executes GV transitions exclusively via explicit `CCF_WP` register writes from PCode/PUNIT.

Two clocking modes:
- **pll_mode = 0 (POR default):** Fast GV — no clock stop during transition, PLL re-locks while frequency changes
- **pll_mode = 1 (survivability):** PLL Crawling — frequency steps ≤ `freq_crawl_delta_f` fuse, requires `pll_mode_ovrden = 1`

### VF Curve Architecture

- 8 VF curves per CBB (one per CCF FIVR) — 6-point interpolated curves
- `UFS_STATUS.CURRENT_VOLTAGE` reports max(8 FIVRs) in U3.13 format
- Freq up: V-first (FIVR ramp → PLL ratio change)
- Freq down: PLL-first (PLL step down → FIVR voltage reduce)
- Out-of-range writes silently clipped to fused [Pm_cap, P0_cap]

## Section 3: Validation Strategy

CBB CCF PM validation requires a three-tier strategy because the feature spans local silicon hardware (GVFSM, PLL, FIVR), CBB-local firmware (PCode), and cross-die firmware (Primecode). No single tier can validate all layers.

> Layer coverage is mapped in §2 — Validation-Tier Layer Claim table identifies which tier validates each stack layer and flags any unclaimed layers as accepted gaps.

### Tier Definitions

| Tier | Environment | Interface | What It Validates |
|---|---|---|---|
| PSS — VP | Simics (virtual platform) | PythonSV → TPMI model | PCode logic (BW heuristic, distress algorithm, ELC decision), BIOS programming flow, safe negative testing (invalid UFS_CONTROL values) |
| PSS — HSLE | Single-die RTL (CBB only) | PythonSV → TPMI RTL | GVFSM state machine correctness, PLL/FLL behavior, VF curve lookup, within-die PCode interaction with silicon |
| PSS — XOS | Both-die RTL (IMH+CBB) | PythonSV → full RTL | Cross-die HPM 0x1b/0x35/0x36 transport, Uniform mode coordination, Primecode Fabric DVFS aggregation |
| FV | Post-silicon NWP | PythonSV → namednodes | Real silicon GV transitions, actual VF curve behavior, real FIVR voltage, real PLL lock timing, PEGA injection, telemetry counter correctness |
| PV | Post-silicon NWP + Ubuntu | intel-speed-select, perf, sysfs | E2E user-visible frequency scaling, driver integration, PMU counter accessibility |

### Why All Tiers Are Needed

- **PSS (VP):** Only safe environment for BIOS negative testing (invalid UFS_CONTROL values) — no silicon damage risk
- **PSS (HSLE):** Only environment with RTL-level GVFSM visibility — can trace individual state transitions
- **PSS (XOS):** Only pre-Si environment with both dies — required for HPM cross-die protocol and Uniform mode
- **FV:** Required for real silicon power/frequency behavior — actual VF curve, actual FIVR, actual PLL
- **PV:** Required for OS/driver layer — intel_pstate, intel-speed-select, PMU events

## Section 5: Risks & Dependencies

### Active Risks

- **Ring C3 (PkgC6) is ZBB on NWP:** Deep CCF power-off state requires PkgC6 which is not exercised on NWP. Ring C3 entry flow and register state can be partially validated, but full power-gate and D2D L1 transition cannot be end-to-end tested until PkgC6 is enabled.
- **Uniform mode requires real cross-die HPM:** Cannot be validated in single-die PSS (HSLE). Requires XOS or FV.
- **D2D PHY upgrade (16→32 GT/s):** NWP D2D PHY speed change increases roundtrip latency to ~72 cycles at 2 GHz — wake event delivery timing differs from DMR. Timing-sensitive tests may need NWP-specific thresholds.
- **VF curve fuse variation:** Different silicon lots may have different fused VF curves. Tests must read actual fuse values rather than hardcoding expected ratios.
- **AutoGV mode spec discrepancy:** Co-Design spec (NWP HAS) describes AutoGV as an available CCF mode where the ring autonomously selects frequency. TCD 22022421183 states AutoGV is **not supported** on CBB (NonAutoGV is the only POR mode). Likely AutoGV exists in silicon but is fuse-disabled or not validated on NWP. **Action:** Confirm with PCode team whether AutoGV is fuse-gated, BIOS-disabled, or architecturally absent on NWP CBB.
- **SBO distress bit-width discrepancy:** Co-Design spec describes ia_distress as 2-bit IA (weak/strong) + 1-bit GT = 3 bits. TCD KB describes `ia_distress[3:0]` (4-bit). **Action:** Verify actual bit-width from RTL or PCode source.

### Accepted Coverage Limitations (by design — no new TCs required)

| Gap ID | Description | Coverage Today | Accepted Rationale |
|--------|-------------|----------------|-------------------|
| **G-1** | Ring C3 full power-gate + PLL restart (PkgC6) | FV partial (entry flow only) | PkgC6 is ZBB on NWP; full Ring C3 validation deferred to platform enablement |
| **G-2** | OS intel_pstate CCF ring frequency visibility | PV only | OS driver integration requires booted OS + intel_pstate — not accessible from PSS or FV |
| **G-3** | Real thermal throttle interaction with CCF GV | FV only | Thermal response requires real silicon power dissipation — model gap in VP/HSLE |
| **G-4** | FIVR voltage accuracy at each VF operating point | FV only | Analog FIVR behavior — cannot be simulated in pre-Si models |

## Section 6: DFX Considerations

- **PEGA B2P injection:** Primary DFx path for isolating CCF GV behavior — bypasses the BW heuristic slow loop for direct ratio control. Available via `pega.pegaPstate(skt, cbb, clrgv=ratio)`
- **TPMI ratio lock:** Setting `UFS_CONTROL.MAX_RATIO = MIN_RATIO` pins CCF to a specific frequency — useful for isolation testing and debug
- **UFS_STATUS readback:** Live ratio and voltage observable without disturbing the system — primary debug tool for frequency validation
- **PLR_DIE_LEVEL:** Performance Limit Reason register — when non-zero, identifies which subsystem is throttling CCF (RAPL, thermal, power limit)
- **PMON CLR counters:** Can be programmed to count ring-scalability events, providing visibility into distress and BW heuristic decisions
- **CCF PMA pm_state CSR:** Direct observability of the CCF idle FSM — confirms idle entry/exit state machine transitions
- **Fast Ring C3 residency counter:** Accumulates cycles in Fast Ring C3 — proves idle state was actually entered and held
